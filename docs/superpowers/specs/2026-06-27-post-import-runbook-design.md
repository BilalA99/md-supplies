# Post-Import Runbook Design

**Date:** 2026-06-27
**Ticket:** P1 · Owner: Sardorbek + Munis + Izzy · Dependency: final Shopify import (assumed complete)
**DoD:** §17.2 runbook passes end-to-end; lead signs off in writing.

## Problem

The final consolidated Shopify import has completed. Before the site can launch, ten post-import verification tasks must pass and be documented. Several concern correctness (collection/category mapping, URL health, sitemap accuracy), several concern SEO (metadata, schema, canonical, robots), one concerns analytics attribution (GTM/GA4 + live purchase), and one concerns compliance (security headers, Sentry, rollback plan). Manual tasks (Merchant Center, GTM live verification, sign-off) are outside the automated scope.

## Approach

Option B — per-domain scripts. One script per concern, each writing to `audit/`. A shell runner sequences them. Manual items get a structured checklist doc.

## Architecture

```
scripts/
  audit-collections.ts          ← existing, re-run (Track 1 / E4)
  check-url-health.ts           ← new (Track 2)
  check-sitemap.ts              ← new (Track 3)
  check-seo-metadata.ts         ← new (Track 4)
  check-security-headers.ts     ← new (Track 5)
  run-post-import-runbook.sh    ← new runner

audit/
  category-nav-audit-report.md  ← existing (regenerated by Track 1)
  URL-HEALTH-REPORT.md          ← Track 2
  SITEMAP-REPORT.md             ← Track 3
  SEO-METADATA-REPORT.md        ← Track 4
  SECURITY-REPORT.md            ← Track 5

docs/
  post-import-qa-checklist.md   ← manual items + sign-off template
```

All automated scripts require `localhost:3000` running and a valid `.env.local` with `SHOPIFY_STOREFRONT_TOKEN` and `SHOPIFY_STORE_DOMAIN`.

## Track 2 — URL Health (`check-url-health.ts`)

**Ticket tasks covered:** Every approved L1/L2 returns 200; 100-PDP stratified QA.

### L1/L2 health check

- Import `getAllowedHandles()` from `lib/category-nav.ts`.
- For each handle, GET `http://localhost:3000/category/{handle}`.
- Record: handle, status code, pass/fail.
- Fail threshold: any non-200 is a blocking failure.

### PDP stratified sample (§17.3)

Stratification: fetch up to 10 products per roadmap L1 category via Storefront API `GET_COLLECTION_PRODUCTS`. Deduplicate by handle. If total > 100, reservoir-sample to exactly 100. If total < 100, use all.

For each sampled product:
- GET `http://localhost:3000/shop/{handle}`.
- Check HTTP 200.
- Parse response body: confirm `<title>` non-empty and `<meta name="description">` non-empty.

Output: `audit/URL-HEALTH-REPORT.md` — table of all category URLs + PDP sample with pass/fail per row, summary counts at top.

## Track 3 — Sitemap Validator (`check-sitemap.ts`)

**Ticket tasks covered:** Sitemap URL-by-URL; redirects/410s on deployed domain.

1. Fetch `http://localhost:3000/sitemap.xml`. Parse all `<loc>` entries.
2. For each loc: GET → record status. Flag non-200 as error; flag 3xx as warning (redirect chain).
3. Read `docs/redirects-ready.json`. For each `{from, to}` entry, GET `http://localhost:3000{from}` — `proxy.ts` (Next.js middleware) intercepts and returns 301 with `Location: /product/{handle}` (note the `/product/` singular rewrite baked into proxy.ts). Confirm status 3xx and Location non-empty. Also check category-level 410s defined inline in `proxy.ts`.
4. Also check `docs/redirect-qa.md` rows where the Pre-launch column is not yet filled — hit each Old path and confirm expected status + Location.

Output: `audit/SITEMAP-REPORT.md` — sitemap URL table + redirect verification table.

## Track 4 — SEO / Schema Validator (`check-seo-metadata.ts`)

**Ticket tasks covered:** Schema/metadata/canonical/robots/OG/internal links validate live.

Pages sampled (fetch HTML from localhost):
- `/` (home)
- `/categories` (hub)
- `/category/gloves` (L1 category — known-live collection)
- `/shop/{first-product-handle}` (PDP — handle resolved from Shopify at runtime)
- `/blog` (blog hub)

Checks per page:
- **Canonical:** `<link rel="canonical" href="...">` present and non-empty.
- **Robots meta:** `<meta name="robots">` either absent (defaults to index,follow) or does not say `noindex`.
- **OG tags:** `og:title`, `og:description`, `og:image` all present and non-empty.
- **JSON-LD:** At least one `<script type="application/ld+json">` block present. PDP must include `@type: "Product"`.
- **Internal links:** Extract all `<a href="...">` with relative or same-origin hrefs. Sample up to 20 per page; GET each → flag any 404.

Output: `audit/SEO-METADATA-REPORT.md` — per-page results table + internal link sample.

## Track 5 — Security Headers (`check-security-headers.ts`)

**Ticket tasks covered:** Security headers, Sentry wired, rollback docs.

1. HEAD `http://localhost:3000/`. Read response headers. Check presence of:
   - `Content-Security-Policy` (or `Content-Security-Policy-Report-Only`)
   - `X-Frame-Options`
   - `X-Content-Type-Options`
   - `Referrer-Policy`
   - `Permissions-Policy`
2. Read `next.config.ts` source — verify `headers()` export contains definitions for each of the above (belt-and-suspenders: confirms headers are in code, not just runtime).
3. Read `instrumentation.ts` — confirm a Sentry DSN is referenced (non-placeholder string).
4. Emit a rollback note section: links to `docs/DEV-02-vercel-setup.md` and notes the revert procedure (revert the security-headers commit + redeploy).

Output: `audit/SECURITY-REPORT.md`.

## Runner (`run-post-import-runbook.sh`)

```sh
#!/usr/bin/env bash
set -e
echo "=== Track 1: Collection Audit ==="
npx tsx scripts/audit-collections.ts

echo "=== Track 2: URL Health ==="
npx tsx scripts/check-url-health.ts

echo "=== Track 3: Sitemap ==="
npx tsx scripts/check-sitemap.ts

echo "=== Track 4: SEO Metadata ==="
npx tsx scripts/check-seo-metadata.ts

echo "=== Track 5: Security Headers ==="
npx tsx scripts/check-security-headers.ts

echo ""
echo "All automated checks complete. Reports in audit/."
echo "Complete manual checklist: docs/post-import-qa-checklist.md"
```

## Error Handling

Scripts must not crash on a single bad URL — catch per-URL errors, mark as `ERROR`, continue. Each script exits with code 1 if any check is a hard failure (non-200 category page, missing required security header, missing canonical on a page), and 0 if all pass (or only warnings).

## Tests

No new tests required — the scripts are one-shot audit runners, not production code. The existing test suite (`vitest run`) must continue to pass after any code changes made as a consequence of audit findings.
